**项目二：投放落地页（C 端高转化广告落地页）**

- **产出 1：首屏性能优化**
  - **[情境]** 投放落地页对加载速度极其敏感，直接影响广告转化效率。当时首屏 FCP 约 2.5s，LCP 约 3.2s，秒开率不足 40%，性能瓶颈亟待解决。
  - **[任务]** 系统性降低首屏加载耗时，提升秒开率。
  - **[行动]**
    - **异步性能数据上报**：采用异步方式上报性能点位数据，性能打点包依赖 Protobuf 序列化库，包体积较大，通过异步发送避免阻塞首屏渲染；
    - **Redis 接口数据缓存**：对落地页核心接口数据进行 Redis 缓存，缩短接口响应时间，降低后端压力；
    - **SSR 产物按需优化**：优化 SSR 构建产物，仅加载当前页面实际使用的组件及其 JS/CSS 资源；按需加载每个页面的 JS 文件，如部分页面加载 Zepto，部分页面不加载，避免冗余资源阻塞首屏。
  - **[结果]** 首屏资源体积与解析耗时明显降低，FCP/LCP 均控制在目标范围内，秒开率显著提升。

- **产出 2：数据看板与漏斗建设**
  - **[情境]** 此前落地页缺乏完整的数据看板，优化效果难以量化，运营同学也无法直观查看广告转化数据。
  - **[任务]** 建设落地页数据看板与广告漏斗看板，补齐数据可视化与监控能力。
  - **[行动]** 新建落地页漏斗看板，补齐从曝光到转化的全链路数据可视化能力；同步建设广告落地页数据看板，整合性能指标与业务转化数据。
  - **[结果]** 看板上线后，运营可直观查看落地页各环节转化数据，辅助日常业务决策；研发侧性能监控与问题排查效率大幅提升，为后续 A/B 实验和日常迭代提供了数据基础。
  - **优化 1：全链路性能优化与秒开率提升**
    - **[情境]** 投放落地页是广告转化漏斗的核心环节，其加载性能直接影响 C 端用户的留存意愿与广告主的投放 ROI。当时核心落地页首屏 FCP 约 2.5s，LCP 约 3.2s，秒开率不足 40%。
    - **[任务]** 系统性降低首屏加载耗时，将 FCP/LCP 控制在 1.5s 以内，并通过数据验证性能提升对业务指标的增益效果。
    - **[行动]** 接入性能打点大盘（监控 FCP / LCP / TTI），优化首屏关键渲染路径：实施代码分割与按需加载、核心资源 preload / dns-prefetch、图片 WebP 化及懒加载；利用 **Redis** 缓存层对落地页核心接口数据进行预热缓存，降低后端响应压力；同时引入高覆盖度单元测试保障优化过程中的稳定性。
    - **[结果]** 核心指标（FCP / LCP）平均提升约 **40%**，1s 内“秒开率”提升近 **15 个百分点**。核心投放接口平均响应时间**缩短 50% 以上**（稳定在 200ms 内），落地页转化率提升约 **8%**（经 A/B 实验验证显著置信）。

- **产出 3：负责包体积实验拿到显著置信实验结果**
  - **[情境]** SEM 投放落地页的包体积直接影响页面加载速度和广告转化成本。老板希望明确「压缩包体积到底能带来多少收益」，以评估是否有必要投入研发资源做包体积治理，但缺乏数据验证，需要严谨的实验来量化收益。
  - **[任务]** 设计并执行包体积 A/B 实验，量化包体积缩减对核心业务指标的影响，最终产出可指导投放预算决策的数据结论。
  - **[行动]** 
    - **第一轮实验失败与排查**：23 年 10 月上线包体积实验，数据无显著结果。作为实验 Owner 主导根因排查，协同前端、客户端、后端、数仓多轮对齐，发现两个核心问题：落地页侧归因的新用户量与站内心跳表存在 gap；部分 SEM 落地页存在漏打点。分别通过增加边缘计算参数归因和替换轻量化打点方案解决数据准确性问题；
    - **第二轮实验异常与口径修正**：重开实验后出现注册率 100%、spam 占比 42% 的异常数据，经多方向排查定位为实验包问题，最终通过修正激活口径（以落地页有访问打点为基准）解决问题，推动实验顺利推进。
  - **[结果]** 重开实验后拿到置信且显著的结果——**包体积越大，激活越少**。基于该结论进一步分析投放成本与包体积的量化关系，直接回应了老板的收益预期。

**项目四：电脑端创作者视频/笔记发布系统（PC Web）**

- **产出：WebAssembly 视频元数据解析方案**
  - **[情境]** PC 端视频发布流程中，需要高精度地前置判断视频横竖屏方向，以适配不同的播放器布局和推荐策略。但传统 Web Canvas 提取视频宽高存在跨浏览器（Chrome / Safari / Edge）、跨编码格式（H.264 / HEVC）的解析差异，准确率仅约 85%，导致部分视频展示异常。
  - **[任务]** 解决行业兼容性难题，实现高稳定性（≥99%）、高准确率的视频横竖屏及元数据（宽高、时长、旋转信息）提取。
  - **[行动]** 创造性地引入 **WebAssembly (Wasm)** 技术，将 FFmpeg 的轻量级元数据解析模块编译为 Wasm，在浏览器端直接读取视频文件头信息（精准提取宽高及 rotation 矩阵），辅以 Canvas 兜底降级策略与完善的数据打点监控，对解析失败 case 进行实时告警。
  - **[结果]** 横竖屏判断准确率从 **85% 提升至 99.5%**，覆盖 Chrome / Safari / Edge 等主流浏览器，为创作者大盘提供了高稳定性、丝滑的发布路径；该方案被团队内多个业务线复用。
